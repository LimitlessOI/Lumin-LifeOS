# Lessons Register

## Purpose

Capture what this thread and the old system already taught us so the blueprint does not pretend to start from zero.

These are not all foundational laws yet.
Some are proven practices.
Some are still hypotheses.
But they must inform the blueprint.

## What we got right

### 1. Builder must not choose work
Strong result.
This removes a major historical source of drift.

### 2. Blueprint is the queue
Work should come only from approved executable blueprint steps.
Not from runners, support-task generators, or Builder improvisation.

### 3. Founder/product ideation is different from internal factory problem-solving
The full 25-idea brainstorm is for founder/product exploration.
Internal factory issues need an escalation ladder, not a giant brainstorm.

### 4. Trust system framing is correct
The real product is justified trust, not code generation volume.

### 5. Salvage should be selective
The old system has valuable infrastructure and valuable lessons.
It should not be treated as a clean base.

## What we got wrong

### 1. Strategic ambiguity was allowed to flow downward
This should have been stopped before BPB.

### 2. Builder-adjacent systems invented work
Support-task generation and patch-plan churn became authority drift in code form.

### 3. Governance outpaced enforcement
The laws became more advanced than the runtime that was supposed to enforce them.

### 4. Memory was too implicit
Important decisions, lessons, and boundaries were not reliably preserved as executable constraints.

### 5. Verification was too shallow
Syntax/existence checks are not enough to prove blueprint compliance.

## Historical failure patterns to carry forward

### Drift patterns
- support-task explosion
- patch-plan recursion
- queue invention
- autonomous reprioritization
- strategic ambiguity pushed into execution

### Execution patterns
- token truncation
- syntax failures
- HTTP 502 / dispatch failures
- unsafe large-file edits
- partial edits that corrupt output

### Governance patterns
- old amendment sprawl as active authority
- human bottleneck caused by unresolved upstream questions
- false confidence from documents that were not enforced

## What this means for the blueprint

The blueprint must include:
- explicit product-development gate
- explicit Builder sandbox
- blueprint-owned acceptance tests
- exact-content verification where output is supposed to be exact
- explicit salvage policy
- explicit blocked-return contract
- explicit dispute/escalation path above Builder

## Current confidence statement

Current confidence should be:
- moderate on architecture direction
- lower on process completeness
- low if Product Development gate is absent

That means:
- do not overclaim
- do not freeze larger missions yet
- prove the gate and the governed loop first
