--## 9 â Devil's Advocate Against the Recommendation

**Against seeding lessons_learned as the first build:**
1. **AM39 Phase 1 is barely proven in production.** If the `lessons_learned` table
is in Neon but the route has never had a real read/write cycle in production,
seeding rows creates false confidence. We'd be calling memory "alive" while the
retrieval path is untested.
2. **The real bottleneck is not memory schema â it is operator retrieval.** Seeded
lessons do nothing if no agent or operator reads them. Without a retrieval path
wired into the builder prompt or `cold-start:gen`, lessons just sit in a table. We'd
have "living memory" that nothing actually lives in.
3. **The proof loop is not closed by seeding alone.** For the
forecastâobserveâcompareâlearn loop to work, the system needs to be *making
forecasts* first. Currently it doesn't track predictions. Seeding past lessons skips
the prediction half and goes straight to "lessons" â which may be superficial
pattern matching, not genuine consequence learning.
4. **SIS1 is PENDING_CONFIRMATION and FPM1 risks are unfixed.** Building
memory layer activation while the lower repair loops are unconfirmed violates the
prove-the-loop rule we just added. The correct order per the rule: confirm SIS1 â
fix FPM1 risks â then activate memory.
**Conclusion of devil's advocate:** Seed lessons_learned, but only after SIS1 is
confirmed and FPM1 quarantine labeling is fixed. And wire at least one retrieval
path (builder context or `cold-start:gen` inclusion) before claiming memory is
"alive."
