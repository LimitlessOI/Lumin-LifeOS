## Mission

Make every AI session **bounded**, **auditable**, and **non-hallucinatory** by forcing reads and writes through explicit lane manifests, generated cold-start packets, and optional strict git gates тАФ without compressing human-readable SSOT markdown into LCL (that remains forbidden for amendments; use manifests for machine pulls).

## Autonomous build charter (blueprint тЖТ house)

**Principle:** Once a feature or product is **fully specified** in SSOT (amendment + manifest + readiness gates + acceptance checks), execution proceeds **without standing approvals**. The human is the architect; agents are the crew. Interruptions are **exceptions only**.

**Adam is pulled in only when all of the following have been tried and failed:**

1. **Blueprint defect** тАФ SSOT contradicts itself, omits a required interface, or fails its own readiness gates (then fix the blueprint first; do not improvise).
2. **Capability gap** тАФ No API, no credential, no legal path, or no safe automation exists yet (then either build the capability or record a scoped decision).
3. **Consensus deadlock** тАФ The council / multi-model consensus protocol returns **unresolved conflict** on a material question (then human chooses; log receipt).
4. **External research exhausted** тАФ For тАЬhow does the industry do X?тАЭ or тАЬwhat broke?тАЭ, agents must run **documented web research** (sources, dates) and attach findings to the lane log **before** escalating.

**Not a reason to interrupt:** stylistic preference, тАЬdouble-check this,тАЭ or low-risk implementation detail тАФ those stay in receipts and CI.

**Continuous improvement:** Scheduled or guarded jobs (Zero-Waste AI per `CLAUDE.md`) may propose SSOT amendments or backlog items; they do **not** silently change production behavior without receipt + verification.

### Separation of judgment and execution (non-negotiable)

No agent may mark its own work as fully trusted. The roles are separated so that no single agent can both do work and verify that work:

- **Builder builds.**
- **Mechanic repairs.**
- **Sentinel verifies.**
- **Mission Control prioritizes.**

An agent that both judges and edits can grade its own homework. This is explicitly forbidden.

### Escalation protocol (when a builder is stuck)

Use this order тАФ do not skip levels, do not send all agents randomly:

1. **Builder stuck** тЖТ Mechanic investigates. Mechanic's output: diagnosis + proposed fix, no implementation yet.
2. **Mechanic proposes fix** тЖТ Sentinel reviews the proposal. Sentinel's output: approve / reject / flag.
3. **Fix affects shared system** (routes, config, migrations, SSOT) тЖТ Council vote required (┬з2.12).
4. **Fix is safe** (narrow repair, receipted, verified) тЖТ Mechanic implements.
5. **Sentinel verifies** the implemented fix against the original diagnosis.
6. **Builder resumes.**

### Full-council trigger (use only when issue is systemic)

All six roles engage only when the problem is platform-wide:
- All queues paused simultaneously
- Repo truth conflict (local vs Railway vs GitHub)
- SSOT conflict between amendments
- Repeated hallucination pattern detected
- Token burn spike with no product throughput
- No product output for 24+ hours despite healthy daemons

### Implementation gate (code, not SSOT)

**Mission Control, Mechanic, and Sentinel are PLANNED.** Before any code is written for any of them:
1. `FAIL_CLOSED` state must be zero тАФ or operator issues an explicit waiver receipt.
2. `PENDING_CONFIRMATION` items must be resolved.
3. ┬з2.12 AI Council vote is required тАФ this is a load-bearing technical architecture decision.
4. A scoped SSOT spec (tasks_file, mandate, accountable_for paths, output format, escalation interface) must exist for the specific agent before implementation starts.

PLANNED agents live in `planned_workers[]` and must not be instantiated by daemon scripts. ## Pre-Flight / Post-Flight Rules

**Pre-flight (before edits):** Read `docs/AI_COLD_START.md`, the lane log from `CONTINUITY_INDEX.md`, and the owning manifest JSON for the task. **If the task touches any SSOT markdown:** read that **entire** file in this session **before** the first edit (`CLAUDE.md` тЖТ **SSOT READ-BEFORE-WRITE**; `SSOT_COMPANION.md` тЖТ **┬з0.5B**).

**Post-flight (before push):** Update lane log + main log pointer + amendment Change Receipts + regenerate `AI_COLD_START.md`.

**Session tags:** Every log entry starts with `[PLAN]`, `[BUILD]`, `[FIX]`, `[REVIEW]`, or `[RESEARCH]`.

**Pre-commit hunk audit (standing engineering rule тАФ added 2026-05-12):** Before every `git commit`, run `git diff --cached` and report all four of the following:
1. Staged files (by name)
2. Hunk categories тАФ what each block of lines does, mapped to a slice name
3. Whether any hunk is outside the current slice
4. Whether the commit is **pure** (single slice) or **mixed-scope** (multiple slices)

**If mixed-scope: STOP.** Do not commit until the operator explicitly approves the combined commit *or* the staged set is split into slice-pure units. This rule was added after commit `5f6d5ebb` bundled SIS1 logic with four previously-unstaged LA1 hunks тАФ functionally correct but causality-breaking for the receipt trail.

**Prove-the-loop rule (standing governance rule тАФ added 2026-05-12):** Do not start the next repair or build slice until the previous slice has **at least one live runtime confirmation** or is **explicitly marked `PENDING_CONFIRMATION`** by the operator in this amendment. A slice is "confirmed" when a real system log event (daemon log, test output, or operator observation) verifies the intended behavior. "Compiles and looks right" is not confirmation. This rule exists because FPM1 was shipped before SIS1 was confirmed live on Forge тАФ functionally fine, but it stacks unverified loops. Violation pattern: building loop N+1 while loop N is still paused in a circuit breaker.
