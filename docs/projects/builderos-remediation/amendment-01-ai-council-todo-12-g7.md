This memo outlines a builder-ready enhancement for the "Ollama system prompt size investigation" task, focusing on the smallest buildable slice to address the identified issue without violating existing constraints.

---

### 1. Blocking Ambiguity or Founder Decision List

The core problem statement "system prompts are too large; needs investigation" presents several ambiguities requiring founder or AI Council decisions before proceeding with implementation beyond initial data gathering:

*   **Target Token Limit:** What is the desired maximum or average token count for Ollama system prompts? Is there a specific percentage reduction target? (e.g., < 2000 tokens per system prompt).
*   **Scope of "System Prompts":** Which specific system prompts are included in the 7,327 avg tokens/call metric? Is this across all Ollama interactions, or a subset?
*   **Acceptable Impact:** What are the acceptable trade-offs for prompt reduction (e.g., slight reduction in model performance, increased latency for multi-stage prompting)?
*   **Prioritization:** Which specific prompts, once identified, should be prioritized for optimization efforts?
*   **Optimization Strategy:** What are the preferred high-level strategies for prompt reduction (e.g., aggressive summarization, dynamic prompt construction, prompt chaining, few-shot examples vs. detailed instructions)?

### 2. Already-Settled Constraints

*   **BuilderOS-only Execution:** All changes must be managed and executed within the BuilderOS governed loop.
*   **No LifeOS/TSOS Impact:** No modifications to LifeOS user features or TSOS customer-facing surfaces are permitted.
*   **Ollama Specific:** The issue is isolated to Ollama interactions and the size of system prompts.
*   **Current Baseline:** The current average system prompt size for Ollama calls is 7,327 tokens.
*   **Investigation Focus:** The immediate next step is investigation, not direct prompt modification.

### 3. Smallest Buildable Next Slice

The smallest buildable next slice focuses on data collection and analysis to inform subsequent optimization efforts. This slice will:

*   **Identify High-Volume Prompts:** Pinpoint the specific system prompts (or prompt templates) that contribute most significantly to the 7,327 average token count.
*   **Quantify Individual Prompt Sizes:** Measure the token count for each identified system prompt instance over a representative period.
*   **Baseline Reporting:** Generate an internal report detailing the identified prompts, their average token counts, and frequency of use.
*   **Hypothesis Generation:** Based on the data, propose initial hypotheses for potential prompt reduction strategies for the most impactful prompts.

This slice explicitly *avoids* any changes to prompt generation logic or LLM interaction code.

### 4. Exact Safe-Scope Files BuilderOS Should Touch First

The following files are within safe scope for this initial investigation slice. These are primarily for internal data collection, logging, and reporting, without altering core functionality.

*   `src/internal/ollama-prompt-logger.js`: A new internal utility to log system prompt content and token counts before sending to Ollama. This would be an *extension* to existing Ollama interaction points, not a modification of their core logic.
*   `src/internal/ollama-prompt-analyzer.js`: A new internal script to process logs generated by `ollama-prompt-logger.js` and generate the baseline report.
*   `docs/projects/builderos-remediation/amendment-01-ai-council-todo-12-g7.md`: This document itself, updated with the plan.
*   `config/internal-logging.json`: Update to enable/configure the new prompt logger.
