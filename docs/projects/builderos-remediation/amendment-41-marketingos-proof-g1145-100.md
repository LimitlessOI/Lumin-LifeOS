Amendment 41 MarketingOS Proof: G1145-100 - Campaign Status SSOT Foundation
This document outlines the proof-closing blueprint note for establishing the `Campaign.status` field within MarketingOS as the Single Source of Truth (SSOT), as per Amendment 41.
---
1. Exact Missing Implementation or Proof Gap
The current system lacks a robust, automated mechanism to verify that the `Campaign.status` field in the MarketingOS primary db is consistently reflected in all integrated downstream marketing platforms (e.g., AdPlatform A, EmailService B) that consume this status, thereby establishing it as the SSOT. Specifically, there is no scheduled reconciliation or real-time validation loop to confirm this consistency.
2. Smallest Safe Build Slice to Close It
Implement a daily scheduled job, `CampaignStatusReconciler`, that performs the following:
1.  Queries all active and recently completed `Campaign` records from the MarketingOS primary db, specifically retrieving their `id` and `status`.
2.  For each campaign, it queries the corresponding campaign status from configured downstream marketing platforms (e.g., AdPlatform A, EmailService B) via their respective APIs.
3.  Compares the `Campaign.status` from MarketingOS with the status reported by each downstream platform.
4.  Logs any discrepancies found, including `campaignId`, MarketingOS status, and the differing status from the downstream platform.
This job will not automatically correct discrepancies but will report them for manual investigation and remediation.
3. Exact Safe-Scope Files to Touch First
-   `src/jobs/CampaignStatusReconciler.js` (new file)
-   `src/config/jobs.js` (add entry for the new job's schedule and configuration)
-   `src/services/marketingPlatformA.js` (add or extend `getCampaignStatus(campaignId)` method)
-   `src/services/emailServiceB.js` (add or extend `getCampaignStatus(campaignId)` method)
-   `src/models/Campaign.js` (ensure `status` field is indexed and accessible for efficient querying)
4. Verifier/Runtime Checks
-   Job Execution Log: Verify that the `CampaignStatusReconciler` job executes successfully daily at its scheduled time. Check for any job-level errors (e.g., connection failures, unhandled exceptions).
-   Discrepancy Log: Monitor the logs generated by `CampaignStatusReconciler` for any reported `status` discrepancies between MarketingOS and downstream platforms. A healthy state implies zero discrepancies.
-   Manual Spot Check: Periodically (e.g., weekly), select 5-10 active, paused, and recently completed campaigns. Manually query their `status` in the MarketingOS db and then directly via the APIs of AdPlatform A and EmailService B. Confirm that the statuses are identical.
-   Service Health: Monitor the health and response times of the `getCampaignStatus` methods in `marketingPlatformA.js` and `emailServiceB.js` to ensure reliable data retrieval from external APIs.
5. Stop Conditions If Runtime Truth Disagrees
-   If the `CampaignStatusReconciler` job fails to execute successfully for 3 consecutive scheduled runs.
-   If the job consistently reports discrepancies for more than 5% of active campaigns over a 7-day period, indicating a systemic synchronization issue.
-   If manual spot checks reveal discrepancies that the automated job did not detect, suggesting a gap in the reconciler's scope or logic.
-   If the `getCampaignStatus` methods in `marketingPlatformA.js` or `emailServiceB.js` consistently fail to retrieve data or return incorrect data due to external API changes, rateLimits, or auth issues, preventing effective reconciliation.
