# Proof-Closing Blueprint Note: AMENDMENT_41_MARKETINGOS - Proof G1025-100

**Source Blueprint:** `docs/projects/AMENDMENT_41_MARKETINGOS.md`
**Signal:** This document — SSOT foundation.

---

### 1. Exact Missing Implementation or Proof Gap

Verification of end-to-end data flow for `UserEngagementEvent` from LifeOS `UserActivityService` to MarketingOS `EventIngestionAPI`, ensuring data integrity, delivery confirmation, and adherence to the schema defined in `AMENDMENT_41_MARKETINGOS.md` section 3.2. This gap specifically addresses the lack of an automated, auditable proof that a user's engagement action in LifeOS successfully triggers a corresponding, correctly formatted event in MarketingOS.

### 2. Smallest Safe Build Slice to Close It

Implement a dedicated integration test case within the `UserActivityService` test suite. This test will:
*   Simulate a `UserEngagementEvent` originating from a LifeOS user action.
*   Utilize the existing `MarketingOSClient` to send this event to a *mocked or staging* MarketingOS `EventIngestionAPI` endpoint.
*   Assert the successful HTTP response (e.g., 200 OK) from the MarketingOS endpoint.
*   Include a unique, traceable correlation ID in the event payload and assert its presence in the mock/staging MarketingOS response or a simulated log.
*   Perform a basic schema validation on the outgoing event payload to ensure it matches the `AMENDMENT_41_MARKETINGOS.md` specification.

### 3. Exact Safe-Scope Files to Touch First

*   `services/user-activity/test/UserActivityService.test.js` (Add new integration test)
*   `services/user-activity/src/integrations/MarketingOSClient.js` (Potentially add a test-specific export or mockable interface if not already present, without altering production logic)
*   `docs/projects/builderos-remediation/amendment-41-marketingos-proof-g1025-100.md` (This document)

### 4. Verifier/Runtime Checks

*   **Automated Test Pass:** Successful execution of the new integration test `UserActivityService.test.js::should_transmit_engagement_event_to_marketingos_with_integrity`.
*   **Staging Environment Observation (Manual/Automated):**
    *   Observation of `UserEngagementEvent` logs in the MarketingOS staging environment (or mock service logs) with the exact matching correlation ID generated by the test.
    *   Confirmation that the ingested event payload in MarketingOS staging matches the expected schema and data values.
*   **Metrics Check:** Monitoring dashboard metrics for `marketingos_event_ingestion_success_total` incrementing by one after the test execution (if applicable to staging).

### 5. Stop Conditions if Runtime Truth Disagrees

*   **Test Failure:** The integration test `should_transmit_engagement_event_to_marketingos_with_integrity` fails (e.g., assertion error on HTTP status, payload mismatch, timeout).
*   **No Event Observed:** No corresponding event logs or metrics are observed in MarketingOS staging (or mock service) within 5 minutes of the test execution, or the correlation ID is missing.
*   **Data Integrity Mismatch:** The event payload observed in MarketingOS staging does not match the expected schema or contains incorrect/missing data fields as per `AMENDMENT_41_MARKETINGOS.md`.
*   **Unexpected Errors:** Any unexpected errors or exceptions are logged by `UserActivityService` during the event transmission in the test environment.
