# Platform Affiliate + Revenue-Share Engine — Design Spec

**Status:** DRAFT / awaiting founder sign-off on money-critical items (see §9)
**Scope:** Platform-wide. ALL sellable products attribute into one shared layer (not per-product). Confirmed by founder: "not just for this, for all of it."
**Model:** eXp-style — modest upfront direct commission + multi-tier downline revenue share, recurring-first.
**Open to everyone:** not creator-gated. Any account can join free (no fee), get a referral link, refer deals, and build a downline. Confirmed by founder: "even non-creators will be incentivized to build a downline… we all win together."

---

## 1. Goal

Let creators (YouTubers, IG creators, educators, any promoter) sign up, get a referral link/code, and earn:

1. **Direct commission** on deals they personally refer, across **every product** (SMOS pack, Site Builder publish/care plan, and any future product checkout).
2. **Downline revenue share** on deals generated by promoters they recruited, across multiple tiers (eXp-style), paid from company revenue — never a fee charged to the downline.

Recurring-first: downline leverage is only meaningful on recurring revenue, so the model is designed to pay a % of each recurring charge for as long as the referred customer stays.

---

## 2. Guardrails (legality + honesty)

- **No fee to join.** Commissions are paid ONLY on real, paid, non-refunded product sales. This is what keeps it a legitimate affiliate/revshare program and NOT a pyramid scheme (the eXp line: revshare comes out of real transaction revenue, joining is free).
- **No recruitment-only earnings.** A creator earns downline revshare only when their recruits make real product sales — never for the act of recruiting.
- **Clawback on refund/chargeback.** Any commission tied to a deal that is later refunded/charged-back is reversed in the ledger.
- **Attribution is evidence-backed.** Every commission row points to a real Stripe `checkout.session` / `payment_intent` id.
- **No overclaim.** Nothing in the creator dashboard is shown as "earned/payable" until the underlying charge has cleared and the refund window policy (§7) is satisfied.

## 3. Data model (new migration)

```
affiliates
  id                uuid pk
  owner_id          text            -- the LifeOS account this creator maps to
  handle            text unique     -- public creator handle
  status            text            -- pending | active | suspended
  sponsor_id        uuid null fk->affiliates.id   -- who recruited them (upline); null = top level
  stripe_account_id text null       -- Stripe Connect connected account (payouts)
  created_at        timestamptz

referral_codes
  id            uuid pk
  affiliate_id  uuid fk->affiliates.id
  code          text unique         -- short code used in ?ref=CODE and /r/CODE
  created_at    timestamptz

attribution_events
  id            uuid pk
  code          text                -- referral code seen
  affiliate_id  uuid fk->affiliates.id
  kind          text                -- click | signup | purchase
  subject_id    text null           -- clientId / marketing_session_id / customer id
  product       text null           -- site-builder-publish | smos-content-pack | ...
  stripe_ref    text null           -- checkout session / payment intent id (purchase only)
  amount_cents  int null            -- deal amount (purchase only)
  created_at    timestamptz

commission_ledger
  id                uuid pk
  affiliate_id      uuid fk->affiliates.id   -- who earns this row
  source_event_id   uuid fk->attribution_events.id  -- the purchase it derives from
  tier              int             -- 1 = direct, 2 = downline L2, 3 = downline L3 ...
  basis_cents       int             -- the deal amount the % applied to
  rate_bps          int             -- basis points (e.g. 3000 = 30%)
  amount_cents      int             -- computed commission
  recurring         bool            -- true if this is a recurring-charge commission
  status            text            -- pending | payable | paid | reversed
  stripe_transfer_id text null      -- set when paid out via Connect
  created_at        timestamptz
```

Sponsor chain is walked via `affiliates.sponsor_id` to compute downline tiers.

---

## 4. Attribution flow

1. **Link:** creator shares `https://<host>/r/<code>` or any product URL with `?ref=<code>`.
2. **Click:** middleware sets a first-party `ref` cookie (90-day) and writes a `click` attribution_event.
3. **Signup / session start:** when a referred visitor starts a product flow, write a `signup` event bound to their subject id (clientId / marketing_session_id).
4. **Checkout:** the referral `code` is injected into the existing Stripe `metadata` of every checkout session (both `smos-pack-checkout.js` and `site-builder-entry-checkout.js` already set a `metadata` object — add `ref_code`).
5. **Verify (payment cleared):** in each `verify*CheckoutSession`, after confirming `paid`, call the shared `recordAffiliateCommission({ code, product, stripeRef, amountCents })` which:
   - writes a `purchase` attribution_event,
   - creates a **tier-1** ledger row for the referring affiliate,
   - walks `sponsor_id` upward creating **tier-2, tier-3** downline rows,
   - marks rows `pending` until the refund window passes (§7).

This reuses the existing checkout metadata + verify hooks — minimal change to product code.

---

## 5. Commission math (defaults — editable in config)

Recommended starting config (`config/affiliate-revshare.js`):

```
direct (tier 1):   3000 bps (30%)
downline tier 2:    500 bps (5%)
downline tier 3:    300 bps (3%)
tiers beyond 3:     0 (expand later, eXp goes to 7)
one-time deals:    tier-1 30% one-time, NO downline (keeps one-time clean)
recurring deals:   tier-1 30% + tier-2 5% + tier-3 3% on EACH recurring charge
```

**Worked example** — creator A recruits B; B refers a $29/mo subscription:
- B earns 30% × $29 = **$8.70/mo** (direct)
- A earns 5% × $29 = **$1.45/mo** (downline tier-2), every month the customer stays.

## 6. Per-product integration points

| Product | Checkout create | Verify hook |
|---|---|---|
| SMOS pack | `services/smos-pack-checkout.js` → add `ref_code` to metadata | `verifySmosPackCheckoutSession` → `recordAffiliateCommission` |
| Site Builder publish + care | `services/site-builder-entry-checkout.js` → add `ref_code` | `verifyPublishCheckoutSession` (replaces ad-hoc "free care month" referral) |
| Future products | same pattern: add `ref_code` to metadata, call shared recorder on verify | |

The existing ad-hoc referral in `site-builder-entry-checkout.js` (credits referrer one free care month) is superseded by this unified ledger.

## 7. Payouts (Stripe Connect) — NEEDS CONFIRMATION

- Creators onboard via **Stripe Connect Express** (`stripe_account_id` stored). Stripe handles KYC + 1099s.
- Refund window: commission rows stay `pending` for **N days** (default 14) after the charge, then flip to `payable`.
- A scheduled job batches `payable` rows per affiliate and issues Stripe **Transfers** to their connected account, flipping rows to `paid` with `stripe_transfer_id`.
- Refund/chargeback webhook → reverse matching ledger rows (`reversed`); if already paid, net against future payouts.
