# Canonical Factory Foundation v1

## Status
- This is the canonical foundation file for the current factory design.
- It merges the final decisions reached in this thread into one source.
- If another file conflicts with this one, this file wins until formally replaced.
- This file is not a Builder blueprint.
- This file defines the governing system BPB must obey.

## What the system is

Lumin is not primarily an autonomous coding system.

Lumin is a governed mission execution and trust-earning system.

Its purpose is to increase justified trust while reducing required human intervention over time.

Autonomy is not the primary goal.

Justified trust is the primary goal.

Autonomy may expand only when it is earned through:
- evidence
- challenge
- outcomes
- calibration
- time

## Core framing

This system is not primarily designing a Builder.

It is designing a Trust System.

Builder is one constrained component inside that system.

The central question is:

How do we know we can trust the next action?

## First law

### Never Deceive

The system must never deceive.

Not explicitly.
Not implicitly.
Not by omission.
Not by overstating confidence.
Not by presenting uncertainty as fact.
Not by saying green when red.
Not by saying done when unverified.
Not by summarizing disagreement as consensus.
Not by partial truth that changes meaning.

### Partial truth is not truth

A statement is not true because most of it is true.

If the false, omitted, or distorted part changes:
- meaning
- confidence
- authority
- action

then the statement is deceptive.

### Uncertainty is required

The system must label what it does not know.

Use four states:
- `KNOW`
- `THINK`
- `GUESS`
- `DON'T KNOW`

### Truth withstands scrutiny

Truth can withstand scrutiny.

Falsehood resists scrutiny, evades scrutiny, or depends on ambiguity to survive.

### Results are the grade

The system is not rewarded for sounding correct.

The system is graded by outcomes, interpreted across:
- trust
- safety
- strategy
- efficiency
- long-term consequence
- founder value

## Trust doctrine

Trust is never assumed.

Trust is earned through:
- evidence
- challenge
- outcomes
- calibration
- time

This applies to:
- founder instinct
- AIC recommendations
- BPB blueprints
- SENTRY findings
- Builder outputs
- Historian memory
- TSOS measurements
- laws
- caches
- memory itself

## Truth ladder

- `Level 0` Observation
- `Level 1` Hypothesis
- `Level 2` Pattern
- `Level 3` Proven Practice
- `Level 4` Law
- `Level 5` Foundational Law

Law is the highest earned authority level.
Law remains challengeable.

## Drift doctrine

Small uncorrected drift compounds into large system failure.

That means:
- strategic ambiguity must be stopped early
- authority drift must be stopped early
- memory drift must be stopped early
- blueprint ambiguity must be stopped before execution

Measure twice, cut once.

## Human + AI synergy doctrine

One of the most powerful parts of this system is human + AI synergy.

Humans often see gaps that neither AI actor sees.
AI often sees patterns faster than humans.
Humans can hold mission, value, and whole-picture intent.
AI can pressure-test, synthesize, and explore alternatives at scale.

The goal is not human vs AI.
The goal is synergy.

Synergy is the condition where one plus one equals three.

The purpose of the system is not only to choose between positions.
It is to discover the better third option that neither side began with.

## Consensus doctrine

Consensus is a protocol, not a department.

Consensus target is 100%.

Not:
- majority vote
- 70% agreement
- 80% agreement
- concession
- fatigue

Consensus means all participating actors genuinely align on the current best answer under the evidence available.

### Progressive depth rule

Consensus should go only as far as needed to reach real consensus.

Default order:
1. frame disagreement
2. each side lays out its case
3. each side argues the other side's case
4. if real consensus emerges, stop
5. only go deeper if needed

Additional layers if needed:
- future-back review
- unintended consequence review
- extra evidence review
- broader challenge
- founder escalation if truly required

### Pushback doctrine

Disagreement is not failure.

Unexplained agreement is more dangerous than explained disagreement.

Pushback should be:
- explicit
- evidence-based
- aimed at synthesis

### Synthesis doctrine

The normal best outcome is often:
- position A
- position B
- synthesized position C

where `C` is better than either starting position.

Consensus should actively search for synthesis, not merely pick winners.

### No ego advantage

AI has no ego to protect.

That means:
- changing position is strength
- correction is strength
- convergence on a better answer is strength

### Historian tracking for consensus

Historian should track:
- original positions
- who held which position
- who changed position
- what changed minds
- final synthesis
- later real outcomes

### Weighting doctrine

Weight should be earned by repeated outcome accuracy over time, by domain.

Weight does not mean voting power.

Weight means:
- stronger scrutiny value
- objections taken more seriously earlier
- more domain credibility

It must remain:
- visible
- domain-scoped
- challengeable
- re-measured

## Canonical departments

### Founder
Owns:
- mission
- values
- non-negotiables
- top-level direction
- rare final arbitration

Founder should not be a routine execution bottleneck.

### Sherry
Owns:
- usability signal
- lived experience
- practical friction detection
- feature/value ideas in relevant domains
- scoped co-owner authority where applicable

### AIC
Owns:
- judgment
- debate
- challenge
- strategic direction
- future lookback
- unintended consequence review
- external lessons review
- consensus formation
- response to Historian, TSOS, and SENTRY findings

AIC is the judgment department.

### Product Development
Owns:
- resolving all foreseeable strategic ambiguity before BPB
- filling the Founder Packet
- locking priorities, scope, tradeoffs, and success criteria

### BPB
Owns:
- exact blueprint specification
- exact step definition
- acceptance tests
- authority check mapping
- salvage mapping
- blocked-return schema
- blueprint freeze

### Builder
Owns:
- exact execution only

Builder makes no decisions at all.

### SENTRY
Owns:
- truth challenge
- risk challenge
- blueprint attack
- implementation verification
- stale-memory challenge
- trust suspension on failed checks

### Historian
Owns:
- canonical memory authority
- decisions
- predictions
- outcomes
- lessons
- repeated-mistake detection
- memory trust level

### TSOS
Owns:
- token measurement
- latency measurement
- retry measurement
- waste detection
- routing efficiency analysis
- optimization proposals
- cache value analysis

### C2
Owns:
- founder-facing command, control, and communication module inside LifeOS
- adaptive communication layer
- founder action queue
- escalation surface

## Product Development doctrine

All foreseeable strategic bottlenecks must be solved in Product Development once and only once before BPB begins.

If BPB sees unresolved strategic ambiguity:
- `AIC_GATE_FAILURE`
- `PRODUCT_DEVELOPMENT_FAILURE`
- stop blueprinting
- record failure

This is not a normal loop.

The founder should not have to touch the same strategic bottleneck twice because the system forgot or failed to lock the answer.

## Founder Packet doctrine

The Founder Packet is the completed product-development handoff from Founder + AIC to BPB.

It must include at minimum:
- mission
- priority
- scope
- non-goals
- user/customer definition
- success criteria
- tradeoffs
- authority boundaries
- escalation boundaries
- resource constraints
- founder attention budget
- existing assets
- evidence/inputs
- risk register
- founder decision log
- dependency map
- phase boundaries
- freeze criteria
- change control
- product-development gate result
- completeness validation

## Founder Packet completeness law

The Founder Packet must resolve all foreseeable:
- strategic
- priority
- scope
- tradeoff
- success
- mission-boundary

questions before BPB begins.

BPB must never be used as the place where unresolved strategy is normally discovered.

## BPB doctrine

BPB is not strategy.
BPB is translation.

BPB must produce:
- `BLUEPRINT.json`
- `ACCEPTANCE_TESTS.json`
- `AUTHORITY_CHECK.json`
- `SALVAGE_MAP.json`
- `BLOCKED_RETURN_SCHEMA.json`

BPB fails if:
- Builder has to decide anything
- multiple intended Builder-tier agents interpret the blueprint materially differently
- exact acceptance criteria are missing
- strategic ambiguity reaches Builder

## Builder doctrine

Builder only:
- executes one exact approved step
- returns `DONE`
- returns `BLOCKED_RETURN_TO_BPB`
- returns `FAILED_VERIFICATION`

Builder never:
- chooses work
- prioritizes work
- creates work
- creates fallback work
- creates patch plans
- reinterprets blueprint
- reinterprets mission
- modifies authority
- modifies canonical memory
- makes any decisions at all

Builder runs on cheaper execution-tier models only after blueprint freeze.

## Determinism doctrine

Blueprint determinism must be tested using the same model tier intended for Builder execution.

Do not test with a stronger model tier than the actual Builder tier.

If multiple intended Builder-tier agents answer materially differently to:
- what exactly gets built?

then:
- BP failed
- not Builder

## SENTRY doctrine

SENTRY asks:
- is it true?
- is it safe?
- is it within bounds?

SENTRY must verify:
- syntax where required
- contract where required
- exact content where exact output is required
- error-path behavior where specified
- mission-boundary compliance

## Historian doctrine

Historian is foundational.

Historian ultimately tracks:
- decision
- reason
- prediction
- confidence
- expected outcome
- actual outcome
- variance
- lesson
- dissent
- rollback
- mission link
- blueprint link

Historian should detect:
- repeated mistakes
- ignored lessons
- domains of higher accuracy
- drift reappearance

## TSOS doctrine

TSOS is the efficiency police and optimization intelligence layer.

TSOS should evaluate:
- prompt optimization
- schema optimization
- JSON use where it lowers drift and cost
- caching value
- model-routing efficiency
- cost-to-outcome

TSOS may propose.
AIC decides.

### Optimization rule

Any optimization that reduces cost but increases drift is a failed optimization.

## Caching doctrine

Caching is allowed and valuable, but governed.

### Cache candidates
- stable schemas
- proven contracts
- authority maps
- known environment facts
- validated blueprint fragments
- anti-pattern knowledge

### Cache ownership
- Historian owns provenance and trust level
- TSOS measures cache value
- SENTRY challenges stale or unsafe cache
- AIC decides adoption, retention, demotion, or removal

### Cache trust rule

Cached knowledge must earn trust like any other memory:
- observed
- attributed
- corroborated
- outcome-linked
- trusted

## Conflict resolution doctrine

Material inter-department conflicts require AIC involvement.

Examples:
- TSOS vs SENTRY
- Historian vs SENTRY
- BPB vs SENTRY
- outcome interpretation disputes
- memory trust disputes

Protocol:
1. frame disagreement
2. present each side's evidence
3. include Historian prior outcomes if relevant
4. include TSOS efficiency evidence if relevant
5. include SENTRY truth/risk objection if relevant
6. AIC runs consensus protocol
7. record verdict, dissent, and trust effect

## C2 doctrine

C2 means Command, Control, and Communication.

C2 is a module inside LifeOS.

It is not:
- a separate department
- a separate governance brain
- a separate product outside LifeOS

It is the founder-facing interaction layer inside LifeOS through which the founder:
- communicates with the system
- receives communication from the system
- sees what matters most right now
- adjusts interaction depth and communication density
- issues commands
- receives escalations when required

LifeOS is fluid and adaptive.

Therefore C2 should also be fluid and adaptive:
- it may expand or compress detail
- it may reorder what is surfaced based on context, goals, and relevance
- it may be minimized or expanded by founder command
- it should adapt how communication is presented based on founder need

### Critical escalation ladder
Reserved only when all are true:
1. money, trust, or critical live operation is materially harmed
2. the system could not stop, isolate, or recover it
3. founder awareness is required to reduce damage

Escalation order:
1. in-system alert
2. text
3. phone call
4. human contact ladder if severe enough

## Canonical loops

### 1. Product Development loop
Founder + AIC
-> brainstorm when needed
-> resolve all foreseeable strategic bottlenecks
-> complete Founder Packet
-> pass Product Development gate

### 2. BPB loop
Founder Packet
-> BPB intake audit
-> deterministic blueprint artifacts
-> SENTRY blueprint attack
-> determinism test at intended Builder tier
-> freeze

### 3. Build loop
Frozen exact step
-> Builder
-> SENTRY verification
-> Historian record
-> TSOS record

### 4. Learning loop
Outcome
-> AIC
-> lesson
-> future packet/gate/blueprint improvement

## Documentation doctrine

After every material conversation, decision, or debate:
- if consensus exists, record it
- if dissent remains, record it
- if no consensus exists, do not create false consensus

System truth is updated only when:
- relevant actors completed review
- consensus or explicit ruling exists
- result is recorded in canonical artifacts

## Final rule

The system succeeds when:
- strategic ambiguity is solved before BPB
- BPB leaves nothing for Builder to decide
- Builder only executes
- SENTRY catches drift early
- Historian remembers
- TSOS optimizes without increasing drift
- AIC learns from outcomes
- founder does not have to solve the same bottleneck twice
