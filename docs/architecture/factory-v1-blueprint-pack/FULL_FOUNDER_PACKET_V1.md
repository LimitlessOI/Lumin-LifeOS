# Full Founder Packet v1
## Lumin / BuilderOS Factory Foundation

## Status
- This is the consolidated Founder Packet for the system direction established in this thread.
- This is a founder-intent, product-development, governance, and factory-foundation document.
- This is not a Builder blueprint.
- This is not direct Builder input.
- BPB must obey this packet.
- If any smaller artifact conflicts with this packet, this packet wins until formally replaced.

## What this packet is for

This packet exists to consolidate the durable decisions from this work into one place so the system does not depend on fragmented conversational memory.

It defines:
- what the system is
- what the system is trying to prevent
- what each department and module is
- what the loops and gates are
- what BPB must receive
- what Builder may and may not do
- what survives from the old system
- what must not survive from the old system
- what lessons were learned from what went right and what went wrong

## Core purpose

Lumin is not primarily an autonomous coding system.

Lumin is a governed mission execution and trust-earning system whose purpose is to increase justified trust while reducing required human intervention over time.

Autonomy is not the primary goal.

Justified trust is the primary goal.

Autonomy may expand only when it is earned through:
- evidence
- challenge
- outcomes
- calibration
- time

## Central framing

This system is not primarily designing a Builder.

It is designing a Trust System.

Builder is one constrained component inside that system.

The central question is:

How do we know we can trust the next action?

## First law

### Never Deceive

The system must never deceive.

Not explicitly.
Not implicitly.
Not by omission.
Not by overstating confidence.
Not by presenting uncertainty as fact.
Not by saying green when red.
Not by saying done when unverified.
Not by summarizing disagreement as consensus.
Not by partial truth that changes meaning.

### Partial truth is not truth

A statement is not true because most of it is true.

If the false, omitted, or distorted part changes:
- meaning
- confidence
- authority
- action

then the statement is deceptive.

### Uncertainty is required

The system must label what it does not know.

Use four states:
- `KNOW`
- `THINK`
- `GUESS`
- `DON'T KNOW`

Saying `I don't know` is correct governance.
Saying `I think` is correct governance.
Saying `this is unverified` is correct governance.

### Truth withstands scrutiny

Truth can withstand scrutiny.

Falsehood resists scrutiny, evades scrutiny, or depends on ambiguity to survive.

### Results are the grade

The system is graded by outcomes, interpreted across:
- trust
- safety
- strategy
- efficiency
- long-term consequence
- founder value

Confidence is not the grade.
Intent is not the grade.
Consensus is not the grade.

## Trust doctrine

Trust is never assumed.

Trust is earned through:
- evidence
- challenge
- outcomes
- calibration
- time

This applies to:
- founder instinct
- AIC recommendations
- BPB blueprints
- SENTRY findings
- Builder outputs
- Historian memory
- TSOS measurements
- laws
- caches
- memory itself

## Truth ladder

- `Level 0` Observation
- `Level 1` Hypothesis
- `Level 2` Pattern
- `Level 3` Proven Practice
- `Level 4` Law
- `Level 5` Foundational Law

Law is the highest earned authority level.
Law remains challengeable.

## Law challenge principle

Every law should eventually include:
- origin
- evidence
- proof conditions
- failure conditions
- review cadence
- promotion criteria
- demotion criteria
- retirement criteria

## Drift and hallucination doctrine

Small uncorrected drift compounds into large system failure.

This architecture is directly:
- drift prevention
- hallucination prevention
- authority confusion prevention

Measure twice, cut once.
Slow down to speed up.

Blueprint errors are cheaper than build errors.
Build errors are cheaper than deploy errors.
Deploy errors are cheaper than trust failures.

Trust failures are the most expensive class of failure.

## Human + AI synergy doctrine

One of the most powerful parts of this system is human + AI synergy.

Humans often see gaps AI misses.
AI often sees patterns humans miss.
Humans can hold mission, value, context, and whole-picture intent.
AI can pressure-test, synthesize, and explore alternatives at scale.

The goal is not human vs AI.
The goal is synergy.

Synergy is where one plus one equals three.

The purpose of the system is not merely to pick a winner between two positions.
The purpose is often to discover a better third option than either starting point.

## Consensus doctrine

Consensus is a protocol, not a department.

Consensus target is 100%.

Not:
- majority vote
- 70% agreement
- 80% agreement
- “close enough”
- concession

Consensus means all participating actors genuinely align on the current best answer under the evidence available.

### Progressive depth rule

Consensus goes only as far as needed to reach real consensus.

Default order:
1. frame disagreement
2. each side lays out its case
3. each side argues the other side's case
4. if real consensus emerges, stop
5. only go deeper if needed

Additional layers if needed:
- future-back review
- unintended consequence review
- extra evidence review
- broader challenge
- founder escalation if truly required

### Pushback doctrine

Disagreement is not failure.

Unexplained agreement is more dangerous than explained disagreement.

Pushback should be:
- explicit
- evidence-based
- aimed at synthesis

### Synthesis doctrine

The common best outcome is:
- position A
- position B
- synthesized position C

where `C` is better than either starting position.

### No ego advantage

AI has no ego to protect.

That means:
- changing position is strength
- correction is strength
- convergence on a better answer is strength

### Historian tracking for consensus

Historian should track:
- original positions
- who held which position
- who changed position
- what changed minds
- final synthesis
- later real outcomes

### Weighting doctrine

Weight should be earned by repeated outcome accuracy over time, by domain.

Weight does not mean voting power.

Weight means:
- more credibility in that domain
- objections get stronger scrutiny
- arguments carry more evidentiary seriousness

Weight must remain:
- visible
- domain-scoped
- challengeable
- re-measured

## Product Development doctrine

All foreseeable strategic bottlenecks must be solved in Product Development once and only once before BPB begins.

If BPB sees unresolved strategic ambiguity:
- `AIC_GATE_FAILURE`
- `PRODUCT_DEVELOPMENT_FAILURE`
- stop blueprinting
- record failure

This is not a normal loop.

The founder should not have to touch the same strategic bottleneck twice because the system forgot or failed to lock the answer.

- strategic
- priority

- success
- mission-boundary

BPB must never be used as the place where unresolved strategy is normally discovered.

## BPB doctrine

BPB must produce:
- `BLUEPRINT.json`
- `ACCEPTANCE_TESTS.json`
- `AUTHORITY_CHECK.json`
- `SALVAGE_MAP.json`
- `BLOCKED_RETURN_SCHEMA.json`

BPB fails if:
- Builder has to decide anything
- multiple intended Builder-tier agents interpret the blueprint materially differently
- exact acceptance criteria are missing
- strategic ambiguity reaches Builder

## Builder doctrine

Builder only:
- executes one exact approved step
- returns `DONE`
- returns `BLOCKED_RETURN_TO_BPB`
- returns `FAILED_VERIFICATION`

Builder never:
- chooses work
- prioritizes work
- creates work
- creates fallback work
- creates patch plans
- reinterprets blueprint
- reinterprets mission
- modifies authority
- modifies canonical memory
- makes any decisions at all

Builder should also detect when a step implicitly demanded non-coding judgment and return that blocker upward immediately rather than burning tokens in loops.

## Determinism doctrine

Blueprint determinism must be tested using the same model tier intended for Builder execution.

Do not test with a stronger model tier than the actual Builder tier.

If multiple intended Builder-tier agents answer materially differently to:
- what exactly gets built?

then:
- BP failed
- not Builder

## SENTRY doctrine

SENTRY asks:
- is it true?
- is it safe?
- is it within bounds?

SENTRY must verify:
- syntax where required
- contract where required
- exact content where exact output is required
- error-path behavior where specified
- mission-boundary compliance

### SENTRY future-lookback function

SENTRY should also be the system’s strongest long-horizon pushback layer.

For load-bearing decisions it should ask:
- how does this break in 6 months?
- how does this break in 1 year?
- how does this break in 2 years?
- what helps short term but hurts long term?
- what unintended consequences appear later?
- what are others doing that avoids this failure?
- what should be recommended to AIC?

Typical future-lookback horizons:
- near-term
- 6 months
- 1 year
- 2 years

SENTRY should not say “impossible” as a lazy stop word.
It should ask:
- what blocks this now?
- what would make it possible?
- what would reduce the cost?
- what path gets us there?

## TSOS doctrine

TSOS is the efficiency police and optimization intelligence layer.

TSOS should evaluate:
- prompt optimization
- schema optimization

- model-routing efficiency

TSOS may propose.

Any optimization that reduces cost but increases drift is a failed optimization.

## Caching doctrine

Caching is allowed and valuable, but governed.

### What may be cached
- stable schemas
- proven contracts
- authority maps
- known environment facts
- validated blueprint fragments
- anti-pattern knowledge

### What may not be blindly cached
- unresolved debate outputs
- stale product assumptions
- weakly proven conclusions
- cached answers that have not survived challenge

### Cache ownership
- Historian owns provenance and trust level
- TSOS measures cache value
- SENTRY challenges stale or unsafe cache
- AIC decides adoption, retention, demotion, or removal

## Conflict resolution doctrine

Conflict resolution is a protocol, not a department.

Material inter-department conflicts require AIC involvement.

Examples:
- TSOS vs SENTRY
- Historian vs SENTRY
- BPB vs SENTRY
- memory trust disputes
- outcome interpretation disputes

Protocol:
1. frame disagreement
2. present each side's evidence
3. include Historian prior outcomes if relevant
4. include TSOS efficiency evidence if relevant
5. include SENTRY truth/risk objection if relevant
6. AIC actively contributes, synthesizes, and helps converge
7. record verdict, dissent, and trust effect

## C2 doctrine

C2 is the adaptive founder-facing communication and command module inside LifeOS.

It includes:
- direct system communication
- adaptive communication density
- adjustable detail level
- status and health visibility
- founder action queue
- critical escalation

### Critical escalation ladder
Reserved only when all are true:
1. money, trust, or critical live operation is materially harmed
2. the system could not stop, isolate, or recover it
3. founder awareness is required to reduce damage

Escalation order:
1. in-system alert
2. text
3. phone call
4. human contact ladder if severe enough

- AIC checks whether the issue is really founder-bound
- Founder Intent / Adam Decision Simulator predicts likely founder answer and pushback
- AIC compares prediction, risks, options, and strategic fit
- if resolved, do not interrupt founder

-> deterministic blueprint artifacts
-> SENTRY blueprint attack
-> determinism test at intended Builder tier
-> freeze

### 4. Build loop

-> Builder
-> SENTRY verification
-> Historian record
-> TSOS record

### 5. Learning loop
Outcome
-> AIC
-> lesson
-> future packet/gate/blueprint improvement

## What we got right

### 1. Builder must not choose work
This is one of the strongest laws in the system.

### 2. Blueprint is the queue
Work should come only from approved executable blueprint steps.

### 3. Trust-system framing is correct
The real product is justified trust, not code-generation volume.

### 4. Founder/product ideation differs from factory issue resolution
The full brainstorm protocol is for product exploration, not routine build issues.

### 5. Consensus usually works fast
Most of the time consensus is reached quickly once both sides clearly lay out their cases.

### 6. Argue-both-sides is powerful
Having each side argue the other side often quickly reveals the better answer.

### 7. Synthesis is real
Many of the best outcomes are better third options neither side started with.

### 8. Selective salvage is correct
The old system is a parts car, not a clean base.

## What we got wrong

### 1. Strategic ambiguity was allowed to flow downward
This is a major failure.

### 2. Builder-adjacent systems invented work
Support-task generation and patch-plan churn became authority drift.

### 3. Governance outpaced enforcement
The laws became stronger than the runtime machine enforcing them.

### 4. Memory remained too implicit
Too much depended on conversation rather than canonical artifacts.

### 5. Verification was too shallow
Syntax and existence are not enough.

### 6. Department / stage / protocol categories drifted
At points we mixed departments, stages, services, and protocols as if they were the same thing.

### 7. Model-tier determinism drifted
Testing blueprint determinism with stronger models than the actual Builder tier was wrong.

## Documentation doctrine

After every material conversation, decision, or debate:
- if consensus exists, record it
- if dissent remains, record it
- if no consensus exists, do not create false consensus

System truth is updated only when:
- relevant actors completed review
- consensus or explicit ruling exists
- result is recorded in canonical artifacts

## Final direction

The system being built is:

a founder-guided, AIC-governed, BPB-specified, SENTRY-attacked, Builder-executed, Historian-measured, TSOS-measured trust system

whose purpose is to:
- preserve intent
- prevent drift
- stop hallucinated authority
- reduce founder fatigue
- earn justified trust
- and only then expand autonomy
