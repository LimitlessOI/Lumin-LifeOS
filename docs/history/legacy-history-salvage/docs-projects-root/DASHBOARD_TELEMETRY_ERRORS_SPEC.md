## Client-Visible Error Envelopes Specification

This document defines the principles and structure for presenting API errors to the LifeOS Dashboard client. The goal is to provide clear, actionable feedback to the user while maintaining privacy and aiding developer debugging.

### 1. API Failure Copy

**1.1 User-Facing Messages:**
- **Generic Fallback:** For most unhandled or unexpected API errors, display a concise, user-friendly message such as: "Something went wrong. Please try again." or "We're experiencing a temporary issue. Your data may not be up to date."
- **Contextual Messages:** Where specific failure modes are known (e.g., network issues, authentication problems, specific API validation errors), provide more precise, yet still user-friendly, guidance.
    - *Example (Network):* "Failed to connect to LifeOS. Please check your internet connection." (as seen in chat error handling)
    - *Example (Authentication):* "Your session has expired. Please refresh the page or log in again."
    - *Example (Chat):* "Lumin is unavailable right now. Please try again later." (as currently implemented)
- **Graceful Degradation:** If a widget or section fails to load data, it should display its specific "Failed to load" message (e.g., "Failed to load MITs") within its boundary rather than a global error.

**1.2 Adam-Only (Developer) Details:**
- Detailed error messages, stack traces, and internal error codes (e.g., from `r.text()` or `d.error`) must be hidden from the general user.
- These details, along with the correlation ID, should be accessible to Adam (the developer/admin) via:
    - Browser developer console (`console.error`).
    - A dedicated debug overlay or mode, activated by a specific flag (e.g., `localStorage.getItem('lifeos_debug_mode')`). This overlay should present the raw API response, including any backend error messages and the correlation ID.

### 2. Retry Mechanisms

**2.1 Automatic Retries (Transparent):**
- For transient network errors or idempotent API calls, the client *may* implement a limited number of automatic retries (e.g., 1-2 retries with exponential backoff) before surfacing an error to the user. This should be transparent.

**2.2 Manual Retry (User-Initiated):**
- For persistent errors, the UI should provide a clear "Retry" button or mechanism within the affected widget or a global notification.
- User guidance should accompany retry failures, suggesting steps like checking network or contacting support.

### 3. Correlation ID Surfacing

**3.1 Definition:**
- Every API request and its corresponding response should include a unique `x-request-id` header (or similar) generated by the server. This ID is crucial for tracing issues across the system.

**3.2 Visibility:**
- The correlation ID must *never* be displayed directly to the end-user in any error message or UI element.
- It must be logged to the browser's developer console (`console.error`) alongside detailed error information.
- It should be made visible in the Adam-only debug overlay/mode (as described in 1.2).

### 4. Privacy and Fail-Closed Principles

**4.1 Data Minimization:**
- Error messages must not expose sensitive user data, internal system architecture, database details, or specific implementation choices.
- Default to providing less information rather than more, especially for unhandled errors.

**4.2 Security:**
- Authentication and authorization failures should be handled securely, prompting re-authentication or indicating lack of permission without revealing specific reasons that could aid attackers.

**4.3 Robustness:**
- The client should gracefully handle malformed or unexpected error responses from the API, preventing UI crashes and displaying the generic fallback message if specific parsing fails.
